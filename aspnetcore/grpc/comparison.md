## <a name="grpc-strengths"></a><span data-ttu-id="2af23-137">gRPC の長所</span><span class="sxs-lookup"><span data-stu-id="2af23-137">gRPC strengths</span></span>

### <a name="performance"></a><span data-ttu-id="2af23-138">パフォーマンス</span><span class="sxs-lookup"><span data-stu-id="2af23-138">Performance</span></span>

<span data-ttu-id="2af23-139">gRPC メッセージは、効率的なバイナリメッセージ形式である[Protobuf](https://developers.google.com/protocol-buffers/docs/overview)を使用してシリアル化されます。</span><span class="sxs-lookup"><span data-stu-id="2af23-139">gRPC messages are serialized using [Protobuf](https://developers.google.com/protocol-buffers/docs/overview), an efficient binary message format.</span></span> <span data-ttu-id="2af23-140">Protobuf は、サーバーとクライアント上で非常に高速にシリアル化します。</span><span class="sxs-lookup"><span data-stu-id="2af23-140">Protobuf serializes very quickly on the server and client.</span></span> <span data-ttu-id="2af23-141">Protobuf のシリアル化では、モバイルアプリのような限られた帯域幅シナリオで重要なメッセージペイロードが小さくなります。</span><span class="sxs-lookup"><span data-stu-id="2af23-141">Protobuf serialization results in small message payloads, important in limited bandwidth scenarios like mobile apps.</span></span>

<span data-ttu-id="2af23-142">gRPC は http/2 向けに設計されており、http 1.x に比べてパフォーマンスが大幅に向上します。</span><span class="sxs-lookup"><span data-stu-id="2af23-142">gRPC is designed for HTTP/2, a major revision of HTTP that provides significant performance benefits over HTTP 1.x:</span></span>

<span data-ttu-id="2af23-192">HTTP API 要求はテキストとして送信され、人間が読み取りや作成を行うことができます。</span><span class="sxs-lookup"><span data-stu-id="2af23-192">HTTP API requests are sent as text and can be read and created by humans.</span></span>

<span data-ttu-id="2af23-193">gRPC メッセージは、既定で Protobuf を使用してエンコードされます。</span><span class="sxs-lookup"><span data-stu-id="2af23-193">gRPC messages are encoded with Protobuf by default.</span></span> <span data-ttu-id="2af23-194">Protobuf は送信と受信が効率的ですが、バイナリ形式は人間が判読できません。</span><span class="sxs-lookup"><span data-stu-id="2af23-194">While Protobuf is efficient to send and receive, its binary format isn't human readable.</span></span> <span data-ttu-id="2af23-195">Protobuf では、 `*.proto`ファイルで指定されているメッセージのインターフェイスの説明を適切に逆シリアル化する必要があります。</span><span class="sxs-lookup"><span data-stu-id="2af23-195">Protobuf requires the message's interface description specified in the `*.proto` file to properly deserialize.</span></span> <span data-ttu-id="2af23-196">ネットワーク上の Protobuf ペイロードを分析し、手動で要求を作成するために、追加のツールが必要です。</span><span class="sxs-lookup"><span data-stu-id="2af23-196">Additional tooling is required to analyze Protobuf payloads on the wire and to compose requests by hand.</span></span>

<span data-ttu-id="2af23-197">[サーバーリフレクション](https://github.com/grpc/grpc/blob/master/doc/server-reflection.md)や[grpc コマンドラインツール](https://github.com/grpc/grpc/blob/master/doc/command_line_tool.md)などの機能は、バイナリ Protobuf メッセージを支援するために用意されています。</span><span class="sxs-lookup"><span data-stu-id="2af23-197">Features such as [server reflection](https://github.com/grpc/grpc/blob/master/doc/server-reflection.md) and the [gRPC command line tool](https://github.com/grpc/grpc/blob/master/doc/command_line_tool.md) exist to assist with binary Protobuf messages.</span></span> <span data-ttu-id="2af23-198">また、Protobuf メッセージは[JSON との間の変換を](https://developers.google.com/protocol-buffers/docs/proto3#json)サポートしています。</span><span class="sxs-lookup"><span data-stu-id="2af23-198">Also, Protobuf messages support [conversion to and from JSON](https://developers.google.com/protocol-buffers/docs/proto3#json).</span></span> <span data-ttu-id="2af23-199">組み込みの JSON 変換は、デバッグ時に、Protobuf メッセージを人間が判読できる形式に変換するための効率的な方法を提供します。</span><span class="sxs-lookup"><span data-stu-id="2af23-199">The built-in JSON conversion provides an efficient way to convert Protobuf messages to and from human readable form when debugging.</span></span>

## <a name="alternative-framework-scenarios"></a><span data-ttu-id="2af23-200">その他のフレームワークのシナリオ</span><span class="sxs-lookup"><span data-stu-id="2af23-200">Alternative framework scenarios</span></span>

<span data-ttu-id="2af23-201">次のシナリオでは、gRPC よりも他のフレームワークをお勧めします。</span><span class="sxs-lookup"><span data-stu-id="2af23-201">Other frameworks are recommended over gRPC in the following scenarios:</span></span>

* <span data-ttu-id="2af23-202">**ブラウザーでアクセス可能な api**&ndash; grpc はブラウザーで完全にはサポートされていません。</span><span class="sxs-lookup"><span data-stu-id="2af23-202">**Browser accessible APIs** &ndash; gRPC isn't fully supported in the browser.</span></span> <span data-ttu-id="2af23-203">gRPC-Web はブラウザーサポートを提供できますが、制限があり、サーバープロキシも導入されています。</span><span class="sxs-lookup"><span data-stu-id="2af23-203">gRPC-Web can offer browser support, but it has limitations and introduces a server proxy.</span></span>
* <span data-ttu-id="2af23-204">**リアルタイム通信をブロードキャスト**する&ndash; grpc では、ストリーミングによるリアルタイム通信がサポートされていますが、登録済み接続へのメッセージのブロードキャストの概念は存在しません。</span><span class="sxs-lookup"><span data-stu-id="2af23-204">**Broadcast real-time communication** &ndash; gRPC supports real-time communication via streaming, but the concept of broadcasting a message out to registered connections doesn't exist.</span></span> <span data-ttu-id="2af23-205">たとえば、チャットルーム内のすべてのクライアントに新しいチャットメッセージを送信するチャットルームの場合、新しいチャットメッセージをクライアントに個別にストリーミングするには、各 gRPC 呼び出しが必要です。</span><span class="sxs-lookup"><span data-stu-id="2af23-205">For example in a chat room scenario where new chat messages should be sent to all clients in the chat room, each gRPC call is required to individually stream new chat messages to the client.</span></span> <span data-ttu-id="2af23-206">[SignalR](xref:signalr/introduction)は、このシナリオに便利なフレームワークです。</span><span class="sxs-lookup"><span data-stu-id="2af23-206">[SignalR](xref:signalr/introduction) is a useful framework for this scenario.</span></span> <span data-ttu-id="2af23-207">SignalR には、永続的な接続と、メッセージをブロードキャストするための組み込みサポートの概念があります。</span><span class="sxs-lookup"><span data-stu-id="2af23-207">SignalR has the concept of persistent connections and built-in support for broadcasting messages.</span></span>
* <span data-ttu-id="2af23-208">**プロセス間通信**&ndash;プロセスは、着信 grpc 呼び出しを受け入れるために、HTTP/2 サーバーをホストする必要があります。</span><span class="sxs-lookup"><span data-stu-id="2af23-208">**Inter-process communication** &ndash; A process must host an HTTP/2 server to accept incoming gRPC calls.</span></span> <span data-ttu-id="2af23-209">Windows では、プロセス間通信[パイプ](/dotnet/standard/io/pipe-operations)は高速で軽量な通信方法です。</span><span class="sxs-lookup"><span data-stu-id="2af23-209">For Windows, inter-process communication [pipes](/dotnet/standard/io/pipe-operations) is a fast, lightweight method of communication.</span></span>
